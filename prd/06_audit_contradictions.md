# Раздел 6. Аудит противоречий и патчи (Omad Arena v1)

Документ фиксирует обнаруженные противоречия между `01-05` и историческими черновиками, а также архитектурные риски, которые нужно закрыть до начала реализации.

## 6.1 Критичные противоречия (P0)

### P0-1: Anti-cheat через `SECRET_KEY_FRONTEND` (некорректная модель безопасности)
- Где найдено:
  - `03_api_contracts.md` (submit hash с `SECRET_KEY_FRONTEND`)
  - `04_backend_architecture.md` (описание той же схемы)
- Проблема:
  - Любой секрет на фронтенде компрометируется.
  - Эта схема не гарантирует достоверность score.
- Патч-решение:
  - Убираем frontend-secret полностью.
  - Вводим серверную модель: `sessionToken (signed) + idempotencyKey + anti-cheat rules + event sanity checks`.
  - Клиент отправляет только факты сессии, решение о начислении всегда за сервером.
- Статус: `PATCHED` (обновлено в `03_api_contracts.md` и `04_backend_architecture.md`).

### P0-2: Неполный контур кошелька
- Где найдено:
  - `02_database_schema.md` хранил баланс только в `User.pointsBalance`.
  - В архитектуре есть `WalletModule` с запретом прямых изменений баланса.
- Проблема:
  - Нет жесткого инварианта и блокировок уровня кошелька.
- Патч-решение:
  - Вводим `Wallet` как отдельную сущность (1:1 к User).
  - Все операции баланса только через `WalletTransaction` + SQL транзакции + row lock.
- Статус: `PATCHED` (обновлено в `02_database_schema.md`).

### P0-3: Исторические черновики конфликтуют с рабочими правилами
- Где найдено:
  - `logic_paradoxes.md` содержит альтернативные правила, часть которых противоречит `01_business_logic.md`.
- Проблема:
  - Новые участники могут принять исторические гипотезы за текущий стандарт.
- Патч-решение:
  - Закрепить, что источник истины: `01-05` + этот аудит.
  - Исторические файлы оставить только как `архив`.
- Статус: `PATCHED` (уточнение добавлено в `backend_prd.md`).

## 6.2 Высокий приоритет (P1)

### P1-1: Не описана идемпотентность ключевых операций
- Где найдено:
  - В API-контрактах отсутствовал обязательный `Idempotency-Key` для purchase/redeem/submit/promo.
- Риск:
  - Двойные списания и повторные редемпшны при ретраях сети.
- Патч:
  - Обязательный `Idempotency-Key` в критичных endpoint.
  - Таблица/кэш идемпотентности.
- Статус: `PATCHED` (обновлено в `03_api_contracts.md` и `04_backend_architecture.md`).

### P1-2: Не закреплена реализация правила «24 часа после удаления аккаунта»
- Где найдено:
  - Правило есть в `01_business_logic.md`, но API/архитектура ранее не описывали enforcement.
- Риск:
  - Абуз попыток через удаление/перерегистрацию.
- Патч:
  - Soft-delete + `IdentityCooldown` по phone hash + проверка в `auth/verify`.
- Статус: `PATCHED` (внесено в `02_database_schema.md`, отражено в `03_api_contracts.md` и `04_backend_architecture.md`).

### P1-3: Не зафиксирован единый timezone-источник для лимитов
- Где найдено:
  - Бизнес-логика: GMT+5 (Ташкент), но не было единого системного сервиса времени.
- Риск:
  - Разночтения при расчете daily limits.
- Патч:
  - Единый `BusinessClockService` с TZ=`Asia/Tashkent`.
- Статус: `PATCHED` (зафиксировано в `03_api_contracts.md` и `04_backend_architecture.md`).

## 6.3 Средний приоритет (P2)

### P2-1: `PromoRedemption` без явного uniqueness
- Риск: повторная активация кода одним пользователем.
- Патч: unique constraint `[userId, promoCodeId]`.
- Статус: `PATCHED` (внесено в `02_database_schema.md`).

### P2-2: Нет формализованного SLA и DLQ для фоновых задач
- Риск: тихое падение задач expire claims/notifications.
- Патч: BullMQ retries, DLQ, alerting.
- Статус: `PATCHED` (детализировано в `04_backend_architecture.md`).

### P2-3: Нет единого реестра решений (ADR)
- Риск: повторное обсуждение уже закрытых технических решений.
- Патч: добавить `ADR` каталог в backend repo на этапе реализации.
- Статус: `PATCHED` (ADR-процесс зафиксирован в `04_backend_architecture.md`, раздел 4.14).

## 6.4 Единые решения, принятые после аудита
1. Источник истины по логике: `01_business_logic.md`.
2. Источник истины по API: `03_api_contracts.md`, но с архитектурными ограничениями из `04_backend_architecture.md`.
3. Часовой пояс бизнес-логики: `Asia/Tashkent` (GMT+5).
4. Попытки: `5` на каждую игру в день.
5. Старт игры списывает только attempt (без списания points).
6. Дневной лимит начислений points: `5000` на пользователя.
7. QR выдачи: TTL `7 дней`, частичная выдача не поддерживается.

## 6.6 Финальные решения по историческим парадоксам
1. Парадокс 1 (первый вход и 1 point за старт): **выбран вариант A** — старт игры не стоит points, расходуется только attempt.
2. Парадокс 2 (удаление/перерегистрация): **выбран вариант B** — в течение 24ч лимит попыток не восстанавливается.
3. Парадокс 3 (частичная выдача): **выбран вариант A** — partial claim запрещен, выдача только целиком.
4. Парадокс 4 (B2B и venue): **выбран вариант A** — жесткая привязка приза к venue network.
5. Парадокс 5 (попытки): **выбран вариант B** — 5 попыток на каждую игру в сутки (до 45 суммарно при 9 играх).
